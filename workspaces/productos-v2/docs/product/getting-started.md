## When To Use ProductOS

Use ProductOS when product work is suffering from fragmented evidence, repeated rewrite loops, weak follow-through, or unreliable stakeholder communication.

## Workspace Shape

The standard workspace has four core surfaces:

- `artifacts/` for structured system-of-record state
- `docs/` for readable documents
- `handoffs/` for execution-facing context
- `feedback/` for product and process learning

## First Run Loop

Start with one bounded loop rather than trying to activate everything at once:

1. capture a real problem in `problem_brief`
2. convert it into `concept_brief` and `prd`
3. generate or refresh the readable docs
4. review the output through the right validation tier
5. package communication only after the source docs are approved

## Expected Outputs

By the end of the first useful pass, ProductOS should give you:

- a decision-ready product frame
- a readable doc set you can share
- a visible validation state
- a clear next action instead of open-ended AI output

## Operating Rules

- structured artifacts remain the source of truth
- readable docs must stay traceable to artifacts
- leadership-facing communication requires validation and PM approval
- unsupported market-facing claims should be deferred, not polished

## Recommended Next Step

Use the starter workspace for new product adoption and use the self-hosting workspace only to keep ProductOS's own `product-overview`, `getting-started`, `positioning`, and `messaging-house` current after every accepted slice.
