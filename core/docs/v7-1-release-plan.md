# V7.1 Release Plan

Purpose: Define the 30/60/90 execution plan for the next external ProductOS release.

## Release premise

ProductOS should not claim broad autonomous PM replacement in `V7.1`.

It should claim one governed PM superpower core:

- messy input to decision-ready product package
- source corpus plus governed external research
- decision package to stakeholder-ready docs and deck
- weekly PM autopilot from live workspace artifacts

## Days 0-30

- keep customer adoption clean by default and block internal/example contamination
- persist discover and operate outputs as first-class workspace artifacts
- align release truth signals so the scorecards, portfolio review, and promotion gate do not disagree
- freeze the external claim set around the four bounded loops only
- define benchmark tracking for speed to insight, rewrite rate, output quality, and operating reliability

Exit criteria:

- the CLI control surface runs in the supported local environment
- discover scoring uses persisted outputs when they exist
- weekly PM autopilot scoring uses persisted outputs when they exist
- no customer-facing flow depends on seeded support assets

## Days 31-60

- implement governed research as a real loop: local corpus, gap detection, research plan, source-tiered external research, contradiction checks, synthesis
- upgrade docs and deck from internal rendering quality to evidence-aware presentation quality
- move weekly PM autopilot onto live queue and artifact inputs only
- make self-improvement scoring more independent from the runtime that assembles the working bundle
- dogfood the four loops on at least three messy-input product scenarios

Exit criteria:

- research outputs tag important claims as `observed`, `inferred`, or `hypothesis`
- docs and deck outputs are usable with minor or no material rewrite
- weekly PM autopilot runs without seeded operator artifacts
- the improvement loop can flag release risks without grading its own generated narration

## Days 61-90

- run the frozen eval suite against the release candidate until status is `passed`
- drive release truthfulness to `healthy`
- run a full Ralph hardening cycle across the candidate
- create one customer-safe demo workspace that proves the four committed loops
- perform one internal red-team review against the release claims
- publish release notes and an upgrade guide with explicit non-goals

Exit criteria:

- frozen eval suite: `passed`
- truthfulness: `healthy`
- no stop-ship item open
- at least one customer-safe workspace passes manual review
- release claims match measured evidence
