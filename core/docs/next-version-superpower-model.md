# Next Version Superpower Model

Purpose: Define the current ProductOS runtime shape for building the next version inside the repo before any web surface exists.

## Decision

The next version remains `CLI + repo` first.

ProductOS should prove the next-version loops inside the current self-hosting workspace before claiming a broader product surface.

## Control Surface

The canonical user surface is the `productos` CLI.

Core commands:

- `productos ingest`
- `productos run discover`
- `productos run align`
- `productos run operate`
- `productos run improve`
- `productos status`
- `productos review`
- `productos export`
- `productos doctor`

## Superpower Build Order

1. messy input to PRD
2. PRD to docs and deck
3. weekly PM autopilot
4. self-improvement loop refresh

## Dogfood Rule

Every major slice should:

1. run in the self-hosting workspace
2. emit a `feature_scorecard`
3. update the `feature_portfolio_review`
4. route sub-`5/5` gaps into the improvement loop

## Current Hardening Order

The current bounded hardening sequence around the promoted `V4.4` baseline is:

1. verify thin-adapter parity across the named repo-contract hosts
2. prove unattended market-refresh safety for the research slice
3. harden broader market-intelligence distribution posture
